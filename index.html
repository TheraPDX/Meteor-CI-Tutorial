Getting Started with Meteor Continuous Integration

These explanatory scripts, videos and slideshows, will get you started easily with Meteor package development, testing, documenting, code style linting and continuous integration

They are designed for small (10GB) Xubuntu 14.04 virtual machines. They have not been tested in any other environment since the idea is to provide clear proven instructions that can be adapted easily to other circumstances. Once you see how it works in one environment setting up for a different one should present few difficulties.

The goal is to help Meteor developers put together a complete continuous integration work environment with :

  • application
  • packages
  • version control
  • logging
  • documentation
  • unit tests
  • end-to-end tests
  • code quality assurance tests
  • deployment

Modus Operandi

Most likely you'll want to flip through the slides to get an overview, then watch the video, then spin up a fresh virtual machine and run through it all. It shouldn't cost you more than an hour or two. The slides change with the keyboard or scroll wheel, but not mouse clicks. There are hyperlinks to deeper details and "Hover Notes" (that are problematic if the slides are also enabled for control by touchscreen taps or mouse click).

Except where indicated, the bash scripts are 'idempotent' -- you can run them repeatedly without adverse consequences.

Prepare the Machine

We start off with a fresh Ubuntu 14.04 LTS virtual machine and install all the platform dependencies. These are global installs requiring 'sudo'. Subsequent stages should not require 'sudo'. The script shown can run to completion without intervention, or stop at intermediate steps to view explanations.

Code Quality - Coding Style and Linting

If you want reusable components, especially if you are working with one or several teams, it's very important to ensure consistent styles and documentation standards from all contributors to all your projects. In this and the next section we'll set up an environment where standards conformance is easy.

The first element is linting with esLint, run in the editor for instant feedback as well as on the command line so conformance can be published and monitored as part of continuous integration.

Code Quality - Automatic Documentation in the Cloud

The second element exploits two powerful tools: jsDoc and "GitHub Pages".

We'll put together a mechanism for easily documenting code files and then publishing them as a browsable manual, automatically, on GitHub.

First steps

To begin, you will need to obtain the tutorial. Open up a terminal window, then copy'n paste these commands into it:

export URI="https://github.com/martinhbramwell"
export TUT="Meteor-CI-Tutorial"
export FILE="archive/master.zip"

echo "Downloading : \n${URI}/${TUT}/${FILE}"
echo "         as : ${TUT}.zip"

wget -O ${TUT}.zip ${URI}/${TUT}/${FILE}
unzip ${TUT}.zip
mv ${TUT}-master ${TUT}
cd ${TUT}

# Ready to run the first script :
#             sudo ./Part01_PrepareTheMachine.sh

The result should look like this.

Then you can run the first script with:

sudo ./Part01_PrepareTheMachine.sh
